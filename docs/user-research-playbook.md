# 图语家真实用户访谈与试用手册

> 这份文档给想帮助图语家接触真实用户的贡献者使用。目标是用一致、尊重、可复用的方式收集真实沟通场景、可用性反馈和词库需求。

## 1. 图语家是什么

图语家是一个面向失语症患者和照护者的公益开源项目。它希望用图片帮助双方完成基本沟通：

- 患者可以点选图片，系统生成候选句并播报给家属。
- 家属可以输入文字或语音，系统把话转换成患者更容易理解的图片序列。
- 没有网络时，核心的选图表达、文字转图、本地图库和历史记录仍应尽量可用。

图语家不是医疗诊断工具，也不承诺治疗效果。当前阶段的重点是验证真实家庭和康复场景里：哪些话最常说、哪些图片最容易理解、哪些流程会让患者或照护者困惑。

### 给一般人的简短介绍

如果你要向不懂技术的人介绍图语家，可以这样说：

> 图语家是一个帮助失语症患者和家人沟通的开源工具。很多失语症患者不是没有想法，而是说不出来，或者听长句子很困难。图语家尝试把日常沟通变成图片：患者可以点图片表达“我想喝水”“我肚子痛”；家属说的话也可以转成一排图片，让患者更容易理解。我们现在想收集真实生活中最常见、最困难的沟通场景，把这个工具做得更实用。

### 给康复老师 / 医护 / 社工的介绍

> 图语家是一个本地优先的图片辅助沟通原型，面向失语症患者和照护者。我们目前不做疗效验证，也不替代临床评估，只做可用性和需求调研：患者和家属在真实场景里最需要表达什么、最难理解什么、哪些图片和交互流程更容易被接受。反馈会匿名整理，用于改进开源项目的核心图库、文字转图片序列流程和无障碍交互。

## 2. 研究目标

本阶段只做用户研究和可用性测试，不做医疗试验。

我们要回答的问题：

- 家属和患者最常遇到的沟通困难是什么？
- 哪些日常话术应该变成测试样本？
- 哪些核心图片必须离线可用？
- 患者更能理解卡通图、真实照片，还是自定义家庭照片？
- 当前网页流程哪里让患者或照护者困惑？
- 手机、平板、电脑哪种设备更符合真实使用习惯？

不要试图回答：

- 图语家是否改善语言康复效果。
- 图语家是否优于某个临床治疗方法。
- 某个患者的病情诊断或治疗建议。

## 3. 可以去哪里找真实反馈

优先从低风险、容易建立信任的渠道开始。

### 校友网络

适合找：

- 康复、护理、医学、心理、社工、教育相关专业校友。
- 在医院、康复中心、养老机构、公益组织工作的校友。
- 愿意帮忙联系科室老师或患者家庭的志愿者。

建议做法：

- 先发一页项目介绍，不直接要求患者试用。
- 先约 20-30 分钟专家访谈，了解真实场景。
- 请对方帮忙判断：哪些场景适合进入患者家庭试用。

### 康复医院 / 言语治疗科 / 神经康复科

适合获取高质量专业反馈，但需要更谨慎。

建议做法：

- 先联系言语治疗师、康复治疗师或科室老师。
- 明确说明这是公益开源原型，不是医疗器械试验。
- 第一轮只做专家访谈和场景收集。
- 如需患者参与，应由科室老师判断是否合适，并遵守机构要求。

### 失语症患者家庭

这是最真实的反馈来源，也最需要保护隐私。

建议做法：

- 尽量通过熟人、康复老师、公益组织介绍。
- 不收集姓名、病历号、住址、电话号码等个人身份信息。
- 不要求患者独立使用未成熟版本。
- 先让家属试用，再决定患者是否参与。

### 公益组织 / 病友群 / 康复社群

适合收集场景样本和设备习惯。

注意：

- 不在公开群里索要病历、照片、录音。
- 可以发布匿名问卷或访谈招募。
- 对公开回复中的个人信息要主动删除或模糊处理。

## 4. 招募话术

### 简短版

> 你好，我们在做一个公益开源项目“图语家”，帮助失语症患者和家属用图片完成基本沟通。现在项目还在原型阶段，我们想了解真实生活里最难沟通的场景，以及现有界面是否容易理解。访谈大约 30 分钟，不涉及医疗诊断，也不会公开任何个人信息。你是否愿意聊一聊，或者帮忙介绍合适的康复老师 / 家属？

### 给校友 / 志愿者

> 我们在推进一个失语症辅助沟通的公益开源项目，需要康复、护理、医学、社工、设计、开发等方向的朋友一起完善。现在最需要的是帮忙接触真实场景：家属平时最难问什么，患者最常想表达什么，哪些图片和流程看不懂。如果你认识相关老师、机构或家庭，希望能帮忙牵线做一次匿名访谈。

### 给康复科室 / 言语治疗师

> 我们希望请您从专业角度帮忙评估一个图片辅助沟通原型。调研不涉及治疗效果判断，只关注真实沟通场景、核心词汇、界面可理解性和使用风险。我们会匿名记录反馈，并把结论用于开源项目改进。若后续涉及患者或家庭参与，会先征求机构和参与者同意。

## 5. 访谈与试用流程

建议每次 30-45 分钟。

### 访谈前

准备：

- 项目介绍页面或 README 链接。
- 本文档中的隐私说明。
- 访谈记录模板。
- 可运行的本地或线上 demo。
- 如果要录音、录像或截图，提前准备单独同意说明。

开场要说明：

- 这是公益开源原型，不是医疗诊断或治疗工具。
- 参与者可以随时停止。
- 默认只做匿名文字记录。
- 不收集病历号、住址、联系电话等敏感身份信息。
- 反馈只用于改进产品、测试样本和核心图库。

### 推荐流程

1. **说明项目和边界（5 分钟）**
   - 介绍图语家是什么。
   - 说明不做医疗判断、不承诺疗效。
   - 确认是否可以匿名记录。

2. **收集真实沟通场景（10-15 分钟）**
   - 问家属或治疗师真实生活里最难沟通的事情。
   - 记录原话，而不是只记录概括。
   - 标注是否紧急、是否必须离线、是否涉及方言。

3. **演示或试用接收端（10 分钟）**
   - 让家属输入 2-3 句真实会说的话。
   - 观察图片序列是否符合预期。
   - 记录家属想删除、替换、重排的地方。

4. **演示或试用患者表达端（10 分钟）**
   - 让参与者尝试选图表达一个需求。
   - 观察是否能找到图片、是否能理解候选句、是否能完成播报。
   - 患者如果参与，应由照护者或专业人员陪同。

5. **总结反馈（5 分钟）**
   - 问最有用的地方。
   - 问最困惑或最不能接受的地方。
   - 问是否愿意后续再试一次。

## 6. 问题清单

不需要每次全部问完。根据对象选择 8-12 个即可。

### 真实沟通场景

- 平时最难沟通的 10 件事是什么？
- 患者最常想表达什么？
- 家属最常说但患者听不懂的话是什么？
- 哪些场景最着急？例如疼痛、上厕所、吃药、呼救。
- 哪些话每天都可能出现？
- 哪些话在医院、康复训练、居家护理里特别常见？

### 词语和语言习惯

- 家属平时说普通话、粤语、方言，还是混合说？
- 哪些词患者经常听不懂？
- 哪些词家属最难猜对？
- 有没有固定问法？例如“要不要去厕所”“是不是这里痛”“想不想喝水”。
- 有没有短句、反复确认、二选一提问的习惯？

### 图片理解

- 患者更容易理解卡通图、真实照片，还是家里自己的照片？
- 哪些图片容易误解？
- 哪些概念必须用自定义照片？例如家人、家、常去地点、常用物品。
- 愿不愿意使用自定义照片？
- 哪些图片只想保存在本地，不希望同步或导出？

### 设备和环境

- 更习惯手机、平板，还是电脑？
- 家属和患者分别会用哪台设备？
- 有没有网络不稳定的场景？
- 是否愿意把网页添加到手机主屏幕？
- 患者能否拖动图片排序？
- 患者能否点击小按钮？需要多大？

### 原型试用

- 看到首页知道怎么开始吗？
- 患者表达流程哪里卡住？
- 接收端图片序列是否看得懂？
- 全屏展示是否合适？
- 哪些文字按钮患者看不懂？
- 没有声音、没有网络、图片未命中时，界面是否让人困惑？

## 8. 隐私和安全边界

### 不要收集

- 姓名、身份证号、病历号、住址、电话号码。
- 患者正脸照片、家庭照片、语音、视频，除非有单独明确同意。
- 详细病情诊断、治疗方案、用药记录。
- 可以识别家庭住址、学校、工作单位、具体病房的信息。

### 可以收集

- 匿名沟通场景。
- 家属或治疗师改写后的日常话术。
- 去身份化的图片需求，例如“常用水杯”“家里厕所”“康复器械”。
- 设备偏好、网络情况、交互困难。
- 对图片和流程的主观反馈。

### 需要单独同意

- 录音。
- 录像。
- 截屏包含真实个人信息。
- 使用自定义照片做内部测试。
- 后续再次联系。

### 公开记录规则

写入 GitHub issue、PR、文档或测试样本前必须：

- 删除人名、地址、联系方式、医院病房号等身份信息。
- 把具体家庭成员改成通用称呼，例如“儿子”“妻子”“照护者”。
- 把具体地点改成通用地点，例如“家”“医院”“康复中心”。
- 不上传患者照片、家庭照片、录音或视频。
- 标注来源类型，例如“家属访谈，匿名改写”。

## 9. 如何把反馈带回 GitHub

### 适合写到 issue 的内容

- 真实话术样本和期望图片序列。
- 核心图库缺口。
- 图片理解问题。
- 患者侧 UI 困惑。
- 离线或设备兼容问题。
- 需要新增的验收标准。

### 推荐关联 issue

- 核心图库和场景词表：[#32](https://github.com/picinterpreter/picinterpreter/issues/32)
- 结构化图库和 Board / PictureSet：[#11](https://github.com/picinterpreter/picinterpreter/issues/11)
- 接收端样本集：[#38](https://github.com/picinterpreter/picinterpreter/issues/38)
- 真机验收清单：[#67](https://github.com/picinterpreter/picinterpreter/issues/67)
- 离线 MVP 验收：[#65](https://github.com/picinterpreter/picinterpreter/issues/65)
- 缺失图片工作流：[#19](https://github.com/picinterpreter/picinterpreter/issues/19)
- 患者侧图标化和无障碍：[#6](https://github.com/picinterpreter/picinterpreter/issues/6)

### 推荐 issue 评论格式

```markdown
## User research note

**Source**
Anonymous caregiver interview / therapist interview / prototype trial

**Scenario**
Eating together outside

**Original utterance**
你要一起出去吃饭吗？

**Expected pictogram sequence**
你 / 一起 / 出去 / 吃饭 / 吗

**Observed issue**
如果系统把“出去吃饭”拆得太碎，患者可能只看到“出去 / 吃”，理解成别的意思。

**Product implication**
“出去吃饭”可能需要作为高频短语保护，或至少保证“出去”和“吃饭”都能稳定命中。

**Privacy check**
No personal names, photos, addresses, or medical identifiers included.
```

## 10. 贡献者注意事项

- 真实用户反馈很珍贵，但不要急着让患者承担测试风险。
- 先访谈家属和治疗师，再安排患者参与。
- 患者参与时，照护者或专业人员应在场。
- 不要把试用过程包装成治疗。
- 不要承诺某个功能一定会被实现。
- 不要让未验证版本用于紧急医疗沟通。
- 每次反馈都要先匿名化，再进入 GitHub。

## 11. 下一步建议

如果你是非技术贡献者，可以从三件事开始：

1. 收集 30-50 条真实家属话术，并写出期望图片序列。
2. 整理 10 个高频场景的必备图片清单。
3. 找 1-2 位康复老师或照护者做 30 分钟访谈。

这些材料会直接帮助图语家的核心图库、接收端 pipeline、离线验收和患者侧 UI 改进。
