# 图语家高风险词清单（V1）

本文档整理的是**词语和图片匹配层面**的高风险词，不是核心词总表。

它回答的是：

- 哪些词即使应该收进核心词库，也特别容易匹错图
- 这些风险有没有外部研究或商业 AAC 结构参考
- 图语家在 V1 应该先加什么护栏

## 怎么读这张表

这里的“高风险”不是指词不重要，而是指它们更容易出现下面几类失败：

- **子串冲突**：短词撞上更长词
- **多义冲突**：同一词在不同场景表示不同概念
- **分词风险**：短语被拆坏
- **否定反转**：否定词被忽略，只匹到正向词
- **领域冲突**：医疗/情绪/食物等不同语义域混淆

外部参考分三种：

- **研究参考**：论文或公开研究告诉我们这类词天然更危险
- **商业 AAC 结构参考**：成熟产品如何通过板块、固定位置、意图分组来避险
- **项目内经验**：图语家已观察到的真实问题

## V1 首批高风险词

| 词语 | 风险类型 | 为什么危险 | 外部参考 | 商业 AAC 常见规避方式 | 图语家 V1 建议 |
|---|---|---|---|---|---|
| `开心` | 子串冲突 / 情绪词 | 容易撞到 `开心果`；情绪词本身也比名词更抽象 | ARASAAC 透明度研究提示形容词/情绪类更脆弱；项目已出现 `开心 -> 开心果` | 情绪词通常固定在 feelings / emotions 板，而不是让自由搜索决定 | 给情绪图加 `semanticDomain: emotions`；给食物图加 `excludeTokens: 开心`；优先固定在 feelings 板 |
| `不开心` | 否定反转 / 情绪词 | 容易被拆成 `不 + 开心`，或者只命中 `开心` | 否定类是 AAC/NLP 常见高风险结构；项目分词规则已专门防 `不开心 -> 开心` | 成熟产品通常把 `sad` / `upset` 作为独立概念，而不是依赖否定组合 | 保留独立概念；不要只靠 `不 + 开心` 组合；禁止对 `不开心` 做 partial 到 `开心` |
| `苹果` | 多义冲突 | 既可能是水果，也可能是品牌/设备入口 | WSD 文献整体支持多义词要做 domain disambiguation | 商业 AAC 往往把食物、物品、科技相关内容放在不同板块中，减少混搜 | 默认 `semanticDomain: food`；如果未来加入设备图，用 `category/domain` 重排；必要时增加 `excludeTokens` |
| `上厕所` | 分词风险 / 场景冲突 | 容易被拆成 `上`、`厕所`、`去厕所` 等多种形式 | 中文 AAC/词库实践里这类固定短语都更适合保留为整体 | 商业 AAC 常把 toilet / bathroom needs 做成独立高频按钮 | 先保留短语；lexicon 把 `上厕所` 归到 `厕所` 概念；患者板固定入口，不依赖开放搜索 |
| `厕所` | 同义词扩散 | `洗手间`、`卫生间`、`上厕所` 都可能绕到这里，既是地点也是需求 | 属于高频需求词；商业 AAC 普遍把这类词固定在 needs / hygiene / toilet 板 | 独立 needs / hygiene / toilet 板；减少自由检索压力 | 保留丰富 alias；固定到 `toilet/hygiene` 板；避免和单纯“地点浏览”混在一起 |
| `头晕` | 分词风险 / 医疗词 | 容易被拆成 `头 + 晕`，导致错图；也可能和泛化“难受”混淆 | 医疗对话 WSD 论文说明医疗域需要专门处理 | 医疗/身体词通常单独成板，不和普通情绪词混排 | 强制 `semanticDomain: medical`；短语优先；不要让 `头` 或 `晕` 的 partial 抢走结果 |
| `难受` | 多义冲突 / 领域冲突 | 既可能是痛、恶心、情绪低落，也可能是泛不适 | 医疗 WSD 和情绪/症状边界都说明这类广义词最难 | 商业 AAC 通过 body-health、feelings、repair 分板，而不是让一个广义词承担所有意义 | 不把它当唯一终点词；更适合映射到候选列表或引导到 body/feelings 板 |
| `痛` | 过宽概念 / 身体词 | 覆盖面太大，和 `头痛`、`肚子疼`、`牙痛` 等具体词关系复杂 | AAC 临床里 pain 常需要部位化表达 | 商业 AAC 里 pain 常和 body-part 或 symptom 一起出现 | 保留 `痛` 核心词，但优先支持 `胸口疼`、`肚子疼`、`头痛` 等更具体概念 |
| `恶心` | 多义冲突 | 中文里既可表示生理恶心，也可表示“讨厌”情绪 | 医疗域词本身就需要上下文保护 | 医疗表达通常进 body/medical 板；情绪表达进 feelings 板 | V1 先默认 `semanticDomain: medical`；如果后续扩展情绪含义，再分概念 |
| `喜欢` | 抽象情绪词 | 抽象、依赖上下文，容易被过度泛化或选到不合适图 | ARASAAC 透明度研究对抽象词有风险启示 | 商业 AAC 通常把 like / want / love 放在稳定核心位 | 作为核心词保留，但尽量用固定板位，不靠模糊搜图 |
| `想` | 高歧义功能词 | 可以是 `want`、`think`、`would like`，不同句子差异大 | 核心词研究表明功能词高频但语义依赖句子 | 商业 AAC 中通常有固定运动位置和稳定板位 | 保留为核心词，不做复杂语义扩张；先保证稳定图和固定板位 |
| `要` | 高歧义功能词 | 可表示想要、需要、将要，特别依赖上下文 | 核心词/句法词在所有 AAC 系统里都依赖位置稳定而非搜索 | 多放在 stable core，而非交给自由检索 | 固定在核心板；不靠 synonyms 无限扩张 |
| `药` | 场景冲突 | 既可能指药物对象，也可能指“我要吃药”这一行为/需求 | 医疗场景常需区分 object vs action | 商业 AAC 会把 medicine 和 take medicine 分开 | 区分 `药` 与 `吃药`；必要时分成 object 和 action 概念 |
| `手机` | 多义邻近 / 动作冲突 | 可能和 `电话`、`打电话`、`联系` 混在一起 | 商业 AAC 往往区分 object 与 action | 物品板和 action/communication 板分开 | 保留 `手机`、`打电话` 两个概念；避免完全共用一张图 |
| `医生` | 人物 / 场景双重身份 | 既是人，也常代表“我要看医生”“医院场景” | 医疗 AAC 板通常把 doctor 作为人物，也有 call nurse / hospital workflow | 商业 AAC 会同时在人物板和医疗板出现 | 允许多板归属；不要只放 people 类 |

## 高风险类别与外部参考的关系

| 风险类别 | 是否有外部研究参考 | 说明 |
|---|---|---|
| 情绪词 / 抽象词 | 有 | ARASAAC 透明度研究表明这类图一般比具体名词更难稳定理解 |
| 医疗症状词 | 有 | 医疗对话转图符的 WSD 论文直接说明医疗域需要专门消歧 |
| 多义词 | 有 | Arasaac-WN、WSD 路线本质上就是为了解决多义词和词义冲突 |
| 中文短语分词风险 | 部分有 | 公开 AAC 文献不一定逐词列举，但中文工程上是明确现实问题 |
| 否定反转 | 部分有 | 更多来自 NLP/AAC 工程经验；图语家当前 matcher 已有否定前缀保护 |
| 子串冲突 | 主要靠项目内经验 | `开心 -> 开心果` 这类问题往往要靠测试集和真实案例积累 |

## 成熟商业 AAC 给图语家的真正启发

商业 AAC 通常**不会公开一份“高风险词清单”**，但它们通过结构设计在规避这类问题：

1. **高频核心词固定位置**
   - `想`、`要`、`喜欢`、`痛` 这类词不靠搜索决定

2. **需求/身体/厕所/修复单独成板**
   - `上厕所`、`头晕`、`恶心`、`叫护士` 这类词放到明确语境里

3. **人物、物品、动作分开**
   - 降低 `医生`、`手机`、`药` 这类词在混合图库里跑偏

4. **允许个体化图片替换**
   - 对于抽象或个体差异很大的词，不强求一张图全员适用

所以图语家不该只问“哪些词危险”，还要问：

- 这个词是不是应该固定在板上，而不是交给搜索
- 这个词是不是应该拆成多个概念
- 这个词是不是应该允许患者偏好图覆盖默认图

## V1 最值得先落地的护栏

按当前阶段，建议优先做：

1. **先给首批高风险词补 `disambiguationHints`**
   - `excludeTokens`
   - `semanticDomain`

2. **把高风险需求词固定到板**
   - `开心`
   - `不开心`
   - `上厕所`
   - `头晕`
   - `恶心`
   - `痛`

3. **把高风险清单转成回归测试**
   - `开心 -> 不能到 开心果`
   - `高兴 -> 不能绕到 开心果`
   - `苹果 -> 默认水果`
   - `头晕 -> 不能拆成 头 + 晕`
   - `不开心 -> 不能命中 开心`

## 结论

高风险词清单和核心词库参考是有关联的，但它们解决的问题不同：

- **核心词参考**解决“哪些词值得优先收进系统”
- **高风险词清单**解决“这些词收进来以后，哪些必须优先加护栏”

对于图语家 V1，最现实的做法不是试图先找到一份现成完美名单，而是：

1. 用研究和商业 AAC 先确定高风险类别
2. 用项目自己的真实问题和测试集补齐逐词清单
3. 先把已知失败案例变成 seed 数据规则和回归测试
